# Такси 1313 — Межгородская такси-платформа Узбекистана

## Overview

Такси 1313 (ранее BuxTaxi) is a production-ready web platform for intercity taxi services in Uzbekistan. It functions as an aggregator connecting passengers, drivers (via PWA), and dispatchers (via web dashboard). The platform facilitates booking, real-time ride tracking, automated dispatch, and robust financial management including commissions and bonuses. Its core purpose is to optimize intercity travel, manage driver operations, and provide comprehensive analytics to lead the intercity transportation market in Uzbekistan.

## User Preferences

The user prefers that the agent works autonomously on tasks but asks for confirmation before implementing any major changes. The user also prefers clear and concise communication.

## System Architecture

The project is built as a monorepo using `pnpm workspaces`.

### Backend (Unified Server)
The backend utilizes `Express 5`, `WebSocket (ws)` for real-time communication, `PostgreSQL` with `Drizzle ORM`, and `JWT` (SHA-256) for authentication. `Web Push API (VAPID)` handles push notifications. Data validation is performed with `Zod v4` and `drizzle-zod`. `Orval` generates API client code. The API covers authentication, rides, drivers, dispatch, tariffs, calls, analytics, and payments, with services for `websocket` management, `autodispatch`, and notifications. **The API server also serves the built frontend static files** — both API and frontend run from the same Express process on port 8080. In production, the deployment build first builds taxi-app (Vite), then bundles api-server (esbuild). The SPA fallback (`/{*splat}` → `index.html`) ensures client-side routing works. No separate static server is needed.

### Core Features

-   **Seat-Based Driver System**: Drivers create rides with 4 available seats, which dispatchers assign to passengers. Drivers see a real-time visual layout of car seats.
-   **District-Based Pricing**: Automatic pricing based on pickup/dropoff districts, including `extraCharge` per district, added to the base route price.
-   **Per-Route Tariff & Options Pricing**: Each tariff class (economy/comfort/business) is independently configurable per route with its own front/back seat prices and its own set of options. `routeOptionsTable` stores per-route, per-tariff options with `tariffClass` column (trunk_small, trunk_large, roof, parcel_s/m/l) — unique constraint on `(routeId, tariffClass, optionKey)`. **Price formula**: `price = (frontSeats × priceFront) + (backSeats × priceBack) + districtFrom + districtTo + sum(options)`. Round-trip: `price * 2 - discount`. `price-estimate` accepts `frontSeats` (0-1) and `backSeats` (0-3) for multi-seat pricing, returns `{ price, priceFront, priceBack }`. Full car = 1×front + 3×back. `passengers` always 1 in ride creation (`POST /rides`). Admin UI: route edit shows tariff cards (Economy/Comfort/Business) with Configure buttons opening nested modal for seat prices + tariff-specific option toggles/prices. Dispatcher UI: tariff dropdown filters options by selected class, clears selections on tariff change. **Driver queue**: `/drivers/by-route` includes online+busy drivers with matching-route rides + truly idle online drivers (no active ride anywhere); returns `carClass` on rides; frontend shows drivers without rides as "Свободен"; filters by `carClass` and uses ±60 min time tolerance. **Seat UI**: clickable seat buttons [1] (front) / [2][3][4] (back) toggle selection via `selectedSeats` Set state. Round-trip auto-selects all 4 seats and disables manual toggle.

-   **Dynamic Routing + Multi-Stop Optimizer**: The multi-stop route optimizer generates all valid permutations of waypoints (with pickup-before-dropoff constraints per passenger enforced via deterministic key-based pairing), pre-filters by haversine distance estimate, then OSRM-routes the top-K candidates to find minimal duration. Detour is allowed if ≤ 15 minutes. Trip stores `waypoints` (jsonb), `routePolyline` (jsonb), `routeDuration`, `routeDistance`, `detourMinutes` — all updated on match/cancel. API: `POST /api/rides/:id/optimize-route` for manual re-optimization. On cancellation, waypoints are cleaned and route is re-optimized via the same optimizer. **Auto-assign to existing trips is DISABLED** — new rides always go through auto-dispatch to drivers who must explicitly accept/buy orders. No automatic passenger assignment to driver trips.
-   **Flexible Route Matching (Detour System)**: Shared route matching engine (`lib/route-match.ts`) used by both auto-dispatch and ride-to-trip matching. Three-tier priority: **exact** (same from+to cities, score=100), **partial** (one city matches + detour within limits, score=40-70), **detour** (both cities differ but route overlaps within limits, score=1-40). Detour calculated as haversine: `(driverFrom→riderFrom + riderFrom→riderTo + riderTo→driverTo) - driverFrom→driverTo`. Direction enforced via `pointProgressAlongLine` (pickup before dropoff along driver route). Configurable via settings: `max_detour_km` (default 50), `max_detour_minutes` (default 40). Scoring incorporates match priority bonus (exact=100, partial=50, detour=20) + detour penalty (extraTimeMin × 2). Stress test achieves 60-80% match rate with 50 drivers / 200 rides across 12 Uzbekistan cities.
-   **Batch Dispatch Engine (Smart Buffering)**: Production ride buffer (`lib/ride-buffer.ts`) collects incoming rides for a configurable window (`batch_window_seconds`, default 90s) before dispatching. Toggle: `batch_dispatch_enabled` (default off, falls back to immediate auto-dispatch). The batch matcher (`lib/batch-dispatch.ts`) uses a multi-pass globally-optimized algorithm: **Pass 1** exact/partial/detour within base limits (50km/40min), **Pass 2** widened limits (80km/64min) for overflow, **Pass 3** ultra-wide (125km/100min) for remaining. Within each pass, rides are sorted scarcity-first (fewer alternatives → higher priority) then by composite score (match quality + driver fill bonus + en-route bonus). Driver priority: prefer drivers already carrying passengers (+20), higher fill ratio (+0-10), more remaining capacity (+5). Stress simulation achieves **70-80% match rate** (55 drivers / 200 rides / 12 cities) with multi-pass batch matching + weighted corridor distribution (Tashkent hub, popular routes weighted by real traffic patterns).
-   **Auto-Dispatch Algorithm**: FIFO queue-based dispatch system (`lib/driver-queue.ts`). Drivers enqueue on going online, dequeue on accept (`markAssigned`), return to queue on ride completion (gated by eligibility: online + balance>0). Queue-based dispatch cycle (`runQueueDispatchCycle`): **First tries route-matched drivers** (drivers with active routes matching the ride's from/to cities), **then falls back to ALL online queue drivers** sorted by distance and score when no route matches found. This ensures orders always reach available drivers even without pre-set routes. **Fill-up priority dispatch**: drivers with fewer free seats get orders FIRST (sorted by `freeSeats ASC`), so nearly-full cars fill up and depart faster. Queue scoring includes `occupiedSeats * 25` bonus. Ignored drivers rotate to queue end (`moveToEnd`), parallel offers via `Promise.allSettled` (fault-tolerant per-candidate). Queue auto-initializes from driver cache on server start. **Pending ride recovery**: `resumePendingDispatches()` runs 10s after server start, finds all pending/offered rides from last 24h without assigned drivers, and restarts dispatch loops for them. Metrics: `queueSize`, `totalEnqueued/Dequeued`, `avgWaitMs`, `skippedDrivers`, `avgAssignTimeMs`, exposed at `GET /api/system/queue`. All queue hooks are best-effort (try/catch) to avoid cascading failures on ride lifecycle. Configurable: `queue_max_distance_km` (500 — intercity radius), `queue_batch_size` (10), `offer_timeout_seconds` (30), `queue_fallback_batches` (3 — number of queue fallback rounds after route-matched drivers exhausted). Full sync: backend `expiresAt`, frontend countdown, looping ringtone sound, vibration every 5s, red pulse animation at ≤5s. **Full-Car Protection**: `findDriversWithActiveRoutes` pre-calculates per-driver occupied seats by finding the route ride (ride with max passengers) for each driver, then uses this count for ALL route-match candidates of that driver. This prevents child rides (attached passengers) from appearing as separate routes with artificially low seat counts. `isDriverStillEligibleForRoute` also queries actual DB passenger counts for queue-path drivers (routeRideId<=0). After accept, `refreshOccupiedSeats()` is called immediately to update the cache. **Smart Driver Ranking**: scoring combines route match priority (exact=100, partial=50, detour=20), rating×5, accept rate×40 bonus, cancel rate×60 penalty, last-active recency, consecutive ignore penalty (×10), hot/cold driver streaks (±50), distance ETA bonus, free seats bonus, detour penalty, **low balance penalty**, **new driver boost**, **idle priority**.

### Revenue Optimization AI Layer
The Revenue AI engine runs as part of the stress monitoring simulation, optimizing revenue without modifying existing systems (stress monitor, alerts, self-healing, optimizer, meta-optimizer all preserved). Key components:
- **Demand-Supply Engine**: Tracks active/busy/idle drivers, pending requests, computes `demand_supply_ratio = requests / available_drivers`
- **Dynamic Pricing (Surge)**: If ratio > 1.2, gradually increases price multiplier (1.0 → 2.0); if demand low, decreases to 0.9 minimum
- **Smart Dispatch Priority**: Ranks drivers by acceptance rate, response speed, completion rate, idle status. Priority weights change based on active strategy mode
- **Driver Utilization**: Idle drivers are prioritized first for ride assignments
- **Revenue Metrics**: `revenue_per_minute`, `completed_rides`, `avg_ride_price` (adjusted by surge), `driver_utilization_pct`, revenue trend tracking
- **AI Strategy Layer**: Three modes tested via meta-optimizer integration — `aggressive` (fast response focus), `conservative` (balanced), `surge_heavy` (maximize per-ride revenue). Auto-switches based on evaluation scores every 20s
- **Logging**: All actions prefixed `[REVENUE AI]` — surge changes, revenue deltas, strategy switches, driver prioritization
- **API Endpoint**: `GET /api/metrics/stress/revenue-ai`
- **Frontend Panel**: `RevenueAIPanel` in StressMonitor with revenue metrics cards, demand/supply gauges, surge multiplier display, strategy evaluation results, expandable driver rankings and activity logs
- **State Reset**: All revenue state resets with simulation reset, preserving clean test isolation

### Revenue AI Production Integration
Production module `lib/revenue-ai-prod.ts` integrates Revenue AI into the real dispatch system without breaking existing dispatch, stress simulation, recovery, or optimizer systems. Key integration points:
- **Dispatch Hook**: In `autodispatch.ts` → `runDispatchCycle()`, after base scoring, Revenue AI reorders eligible drivers using `reorderDriversForRevenue()` — prioritizes idle drivers, high acceptance rate, high completion rate, fast responders. Mode-specific weighting (aggressive/conservative/surge_heavy)
- **Dynamic Pricing**: Revenue AI surge multiplier applied on top of existing surge in both `POST /rides` (creation) and `POST /price-estimate` endpoints via `applySurgeToPrice()`. Safety clamp: min 0.9, max 2.0
- **Strategy-Based Dispatch**: `aggressive` → no offer limit override + 10s timeout, `conservative` → 3 max offers + 20s timeout, `surge_heavy` → no offer limit override + 8s timeout
- **Real-Time Feedback Loop**: `recordDriverOffer()` on each offer sent, `recordDriverAccept()` with response time on accept, `recordDriverReject()` on reject, `recordRideCompleted()` with price on completion. All feed back into Revenue AI cycle via `updateProdCycle()`
- **Safety Limits**: Surge clamped to [0.9, 2.0]. Low-ranked drivers get 15% floor guarantee via `MIN_DRIVER_FLOOR_PCT` — worst-ranked driver bumped to position 3 to prevent starvation
- **Feature Flag**: `revenue_ai_enabled` setting (default `true`). Also toggle via `POST /api/metrics/stress/revenue-ai-prod/toggle`
- **Logging**: All actions prefixed `[REVENUE AI PROD]` — driver reordering, surge application, strategy switches
- **API**: `GET /api/metrics/stress/revenue-ai-prod` (state + logs), `POST /api/metrics/stress/revenue-ai-prod/toggle` (enable/disable)
- **Frontend Panel**: `RevenueAIProdPanel` in StressMonitor with revenue/min, rides, surge, utilization cards, mode/ratio display, toggle button, expandable logs
- **Safety Guard Layer**: Production safety system preventing harmful AI decisions:
  - *Fair Distribution Guard*: Drivers starved > 5 min without offers are force-included in candidate pool. Counter: `fairness_applied`
  - *Surge Protection*: Max surge increase rate limited to 0.15/min. Prevents sudden price spikes. Logged: `[SAFETY] surge rate limited`
  - *Strategy Safety Check*: Before applying new strategy, simulates impact on acceptance_rate and utilization. Blocks if predicted acc < 30% or util > 95%. Counter: `safety_blocks`. Logged: `[SAFETY] strategy blocked`
  - *Kill Switch*: Auto-disables Revenue AI if acceptance_rate drops > 20% or revenue_rate drops > 15%. Logged: `[SAFETY] KILL SWITCH`. Frontend shows animated `KILL SWITCH` badge
  - *Shadow Mode*: `revenue_ai_shadow_mode` — logs all AI decisions without applying them. Compares AI vs default dispatch. Toggle: `POST /api/metrics/stress/revenue-ai-prod/shadow`. Frontend has `Shadow` / `Exit Shadow` button + expandable shadow logs
  - *Diversity Injection*: 25% of driver selection always randomized to prevent monoculture bias. Counter: `diversity_injected`
  - *Safety API*: `GET /api/metrics/stress/revenue-ai-prod` returns `safety` object with all counters. Frontend: expandable Safety Guard section with fairness/diversity/blocks/starved driver stats

## Performance & Monitoring
-   **Node.js Memory Limit**: `--max-old-space-size=512` in start script. Prevents unbounded heap growth.
-   **DB Pool**: `pg.Pool` with `max: 20`, `idleTimeoutMillis: 30000`, `connectionTimeoutMillis: 5000`, `statement_timeout: 10000`.
-   **Photo Compression**: Sharp-based compression on upload — resize to max 1280px, JPEG quality 80 (mozjpeg). Original deleted if optimized version is smaller. Applied to both photo-control and driver profile uploads.
-   **Polling Intervals**: All dispatcher pages standardized to 15s (Overview, Drivers, Map, Archive, SystemMonitor, StressMonitor 8 intervals). Driver: Orders refresh 10s, queue 10s, marketplace 15s.
-   **Global Map Caps**: `driverProdStats` capped at 500 entries with 24h TTL + periodic cleanup every 5min + LRU eviction. `prodStrategyHistory`/`prodSnapshots` capped at 30. `prodLogs`/`shadowLogs` capped at 100/50.
-   **CRM Query Optimization**: Transactions query uses column selection (id, driverId, type, amount, createdAt) + WHERE filter instead of SELECT *.
-   **WebSocket Optimization**: Duplicate connection enforcement — when a user authenticates, old connections with same userId are closed with code 4001. Heartbeat every 30s terminates dead sockets. Immediate cleanup on close. Max 3 connections per user.
-   **Ultra Fast Dispatch Engine**: Yandex/Uber-level dispatch with: in-memory driver cache (synced every 30s + real-time WS location updates), new scoring formula `score = (distance * 0.4) + (idleTime * -0.3) + (rating * -0.2) + (cancelRate * 0.3) + (groupPriority * -0.4)`, dynamic radius expansion (2km → 4km → 6km → 10km → 20km → 50km → 80km), parallel offers to top 3 drivers simultaneously (15s timeout), time tracking (time_to_assign, retries_count), busy driver penalty, idle driver boost. Target: <500ms assignment time.
-   **Response Caching**: In-memory TTL cache (`lib/perf-cache.ts`) for GET endpoints. `cachedEndpoint(ttlMs)` middleware stores JSON responses with X-Cache HIT/MISS headers. Auto-cleanup every 30s. Registered with Memory Guardian for pressure-based eviction.
-   **RPS Tracking**: Global request-per-second counter with 1s buckets over 60s sliding window. Per-endpoint RPS tracked. `rpsMiddleware` wired in `app.ts` before all routes. Top 10 endpoints reported.
-   **Slow Query Logging**: Queries >100ms logged to console + stored in rotating buffer (max 100). Last 20 shown in perf-stats endpoint. Threshold: 100ms in both `lib/db` (trigger) and `perf-cache.ts` (filter).
-   **System Monitoring Dashboard** (`/dispatcher/system`): 4 metric cards (CPU, RAM with Used/Cache/Free breakdown, Disk, Uptime), SVG mini-charts (history capped at 50 points), service status indicators, color-coded alerts. 15s auto-refresh.
